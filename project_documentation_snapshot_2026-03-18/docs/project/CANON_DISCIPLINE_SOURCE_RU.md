Каноническая дисциплина реализации проекта BDC <Авакумов-Коканд>

Ниже - жёсткий "канон" (не рекомендации, а правила), который превращает проект в воспроизводимый исследовательско-инженерный конвейер. Любое отступление считается дефектом процесса и требует исправления так же, как дефект кода.

0) Главный принцип канона

Никаких утверждений без артефактов и метрик.
Любая фраза вида "работает", "лучше", "масштабируется", "снижает" легитимна только вместе с:

протоколом эксперимента,

фиксированными сидами,

результатами (сырые данные),

статистикой,

и ссылкой на commit/тег.

1) Неприкосновенные аксиомы (Non-Negotiables)

Воспроизводимость по умолчанию
Один и тот же commit + те же сиды + та же среда запуска ? те же результаты (в пределах допусков).

Kill-criteria - закон
Если kill-критерий сработал, направление закрывается или переопределяется. Запрещена рационализация "почти получилось".

Разделение контуров

qcore (логика)

memory (память)

env (среда)

agent (контроллер/геном/VM)

evolution (эволюция/мутация)
Никаких "смешанных модулей" и скрытых зависимостей.

Запрет на "магические" источники смысла
Число ? - только детерминированный поток/эталон/сидирование. Никакой семантики "обучения на ?".

Никакого железа до закрытия TRL5
Любая аппаратная активность до TRL5 считается распылением и нарушением канона.

2) Единый язык доказательств (Evidence Standard)
2.1. Пакет доказательства для любого утверждения

Каждое утверждение "H1 подтверждена", "память >99.9%" должно иметь:

EXPERIMENT_SPEC.md (цель, гипотеза, метрики, пороги, бейзлайны, ограничения)

SEEDS.md (все сиды, схема разбиения потоков)

RUN_COMMANDS.md (точные команды запуска)

RESULTS/*.csv (сырые результаты по всем сидам)

ANALYSIS.ipynb или analysis.py (детерминированный расчёт метрик)

REPORT.md (выводы + ссылки на файлы + CI-артефакты)

commit hash + тег релиза эксперимента

2.2. Статистика - обязательна

минимум N=30 независимых сидов, если не оговорено иначе,

доверительные интервалы,

заранее зафиксированные пороги (не "подбираются" после просмотра данных).

3) Канон управления изменениями (Change Control)
3.1. Запрещены "мелкие правки без протокола"

Любая правка, влияющая на результат эксперимента, обязана сопровождаться:

обновлением EXPERIMENT_SPEC.md (если меняется методика),

либо записью в DECISIONS/ADR-XXXX.md (если меняется принцип).

3.2. Любое изменение имеет класс

R0 (refactor): не меняет численные результаты (должно быть подтверждено регресс-прогоном)

R1 (method): меняет метод/метрики/порог - требует новой версии эксперимента

R2 (hypothesis): меняет гипотезу/kill-criteria - требует новой редакции дорожной карты

4) Канон репозитория (структура и артефакты)

Рекомендуемая структура (каноническая):

qcore/ - реализация логики + таблицы истинности + тесты

memory/ - индекс, целостность, компрессия (опционально), random access

env/ - генерация среды, сценарии, шум, препятствия

agent/ - контроллер, геном, VM, conflict_flag-поведение

evolution/ - селекция, мутации, ограничения на "читерство"

experiments/

exp_0001_h1_calibration/

exp_0002_h3_regeneration/

exp_0003_h2_pistream_memory/

decisions/ - ADR-документы (Decision Records)

results/ - результаты (в git - только агрегаты; сырые - в релиз-артефактах CI или отдельном хранилище)

docs/ - дорожная карта, канон, определения, семантика

scripts/ - 1-кнопочный прогон, сборка результатов, упаковка отчёта

Обязательные файлы-эталоны:

CANON.md - этот документ (дисциплина)

SEMANTICS.md - семантика T/F/MY/MN + conflict_flag

KILL_CRITERIA.yaml - единый файл порогов

SEED_POLICY.md - политика сидов/потоков

REPRODUCIBILITY.md - как воспроизвести всё с нуля

VERSIONING.md - схема версий экспериментов и TRL

5) Канон сидирования и PiStream (жёсткая политика)
5.1. Разделение потоков

Запрещён один и тот же поток для разных подсистем. Минимум:

ENV_PISTREAM - генерация мира

INIT_PISTREAM - начальные состояния

NOISE_PISTREAM - шум сенсоров/среды

MUT_DECISION_PISTREAM - где мутировать

MUT_MAGNITUDE_PISTREAM - насколько мутировать

QUERY_PISTREAM - какие фрагменты памяти запрашивать (random access)

5.2. Фиксация преобразования ?  четверичный поток

В SEED_POLICY.md фиксируется единый метод:

способ получения цифр (base-10 или base-16),

способ перевода в base-4,

правила выравнивания длины,

контрольные хэши потоков.

Запрет: менять метод конвертации ? по ходу экспериментов без новой версии и ADR.

6) Канон TRL-гейтинга (переход только через чек-листы)
TRL1-2 (qcore готов)

Чек-лист перехода:

qcore покрыт unit-тестами, включая конфликтные случаи,

существует таблица истинности/референс,

conflict_flag реализован и тестируется,

есть регресс-прогон, который не зависит от случайности.

TRL3 (Paramecium MVP)

Чек-лист:

1-кнопочный прогон scripts/run_exp_*.sh|ps1

N?30 сидов, все сиды зафиксированы

есть минимум 2 бейзлайна

доказан анти-чит: ограничения на долю MY/MN, штрафы энергии/сканирования, штраф размера генома

критерии пройдены либо направление закрыто по kill-criteria

TRL4 (память)

Чек-лист:

exact recall ? 99.9% на зафиксированной длине

random access latency в целевом классе (зафиксировано заранее)

целостность через checksum/ECC с отчётностью ошибок

тесты на дрейф после обучения/эволюции

TRL5 (интеграция)

Чек-лист:

платформа стабильна (повторяемость прогонов, отсутствие "флуктуаций" от окружения)

пилотная задача с неопределённостью показывает измеримый выигрыш по заранее заданной метрике

документированная стоимость вычислений/ресурсов (даже в симуляции)

7) Канон "анти-самообман" (защита от красивых, но пустых побед)

Запрет на единственный показатель
Нельзя объявлять успех по одной метрике. Минимум:

полезность/utility,

калибровка/ошибка,

стоимость (время/шаги/энергия-прокси),

устойчивость (повреждения/шум/дрейф).

Запрет на скрытую подгонку
Любое изменение порогов/условий после просмотра результатов - недействительно без новой версии эксперимента.

Чёрный ящик запрещён
Каждая часть (qcore, память, VM, регенерация) должна иметь "минимальное объяснение механизма" в docs/ и тесты, подтверждающие именно механизм, а не случайную удачу.

8) Канон журналирования и отчётности (минимальный бюрократический слой)
8.1. Еженедельный цикл (ритуал дисциплины)

Каждую неделю обязателен пакет:

WEEKLY_STATUS.md (что сделано, что доказано, что провалено, что изменено)

ссылка на релиз эксперимента или причина отсутствия

список открытых рисков (3-7 пунктов) и меры

8.2. Decision Records (ADR)

Любое принципиальное решение оформляется как decisions/ADR-000X-*.md:

контекст,

решение,

альтернативы,

последствия,

критерий пересмотра.

9) Канон безопасности и IP (минимум, но строго)

Все исходники и артефакты экспериментов помечаются лицензией/политикой доступа.

Любые внешние данные (если появятся) проходят чек:

источник,

разрешения,

воспроизводимость,

отсутствие утечки приватного.

Никаких "случайных" зависимостей: список зависимостей фиксируется (lockfile обязателен).

10) Определение нарушения канона (и санкции процесса)

Нарушение канона - это, например:

отсутствие сидов/протокола при выводах,

изменение конвертации ? без ADR и новой версии,

"подкрутка" порогов после результатов,

аппаратные разговоры/работы до TRL5,

смешивание контуров и скрытые зависимости.

Санкция:

результат считается недействительным,

создаётся issue "Canon Violation",

делается "исправляющий релиз" с восстановлением воспроизводимости.

Минимальный набор артефактов, который должен существовать всегда

CANON.md

KILL_CRITERIA.yaml

SEED_POLICY.md

SEMANTICS.md

experiments/exp_0001.../EXPERIMENT_SPEC.md (хотя бы один)

scripts/run_all.ps1|sh (1-кнопочный прогон)

REPRODUCIBILITY.md

Мудрая фраза: настоящий прогресс начинается там, где нельзя обмануть ни себя, ни эксперимент.
