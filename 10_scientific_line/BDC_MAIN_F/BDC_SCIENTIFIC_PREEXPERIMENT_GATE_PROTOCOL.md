# BDC Scientific Pre-Experiment Gate Protocol

**Date:** 2026-03-19  
**Status:** ACTIVE PROTOCOL

## Purpose

This document defines the mandatory pre-experiment gate for the continuing scientific BDC line.

The rule is simple:

**New scientific cycles must use `BDC Designer` first as the narrowing, evidence-discipline, and decision-gate layer before direct experiment execution begins.**

This protocol exists because `BDC Designer` is now the first successful operational byproduct of the original BDC research line, and the project should not allow the scientific track to drift back into pre-designer ambiguity.

## Why this gate now exists

Historical BDC repeatedly suffered from:
- architecture inflation before bounded evidence existed,
- weak separation between hypothesis and implementation,
- weak separation between measured and inferred claims,
- and phase drift caused by enthusiasm rather than formal gate logic.

`BDC Designer` proved useful precisely in those failure zones.

Therefore the correct project response is not to admire that success in isolation, but to route the main scientific line through it.

## Scope

This protocol applies to:
- new reboot phases,
- new scientific experiment packages,
- new environment/task proposals,
- new mechanism-validation cycles,
- and any proposed phase transition in the scientific BDC line.

This protocol does **not** replace direct experiments.

It governs the stage that must happen before direct experiment execution.

## Mandatory pre-experiment questions

Before a new scientific cycle starts, the project must be able to answer:

1. What exact hypothesis is being tested?
2. What is the bounded decision gate for this cycle?
3. What evidence is already measured, historical, inferred, or missing?
4. What false next steps must be explicitly excluded?
5. What would count as:
   - `PASS`
   - `FAIL`
   - `REMAIN_IN_PHASE`
6. What is the smallest useful execution scope?

If those questions are not yet explicit, the scientific cycle is not ready to start.

## Required use of `BDC Designer`

Before direct execution begins, `BDC Designer` should be used to produce a bounded pre-experiment package with at least the following outputs:

1. **Evidence state summary**
- what is measured
- what is historical prior
- what is inferred
- what is missing
- what is contradicted

2. **Decision-gate framing**
- what decision is actually being made
- what decision is *not* being made
- what counts as a justified next cycle

3. **Measurement-gap output**
- what evidence is still missing
- which missing measurements block the next decision

4. **False-direction control**
- what not to do next
- what tempting but unsupported expansions must be excluded

5. **Minimal execution scope recommendation**
- what is the narrowest honest cycle that can answer the question

## Minimum required pre-experiment artifacts

Every new scientific cycle should produce, before execution:

1. a cycle packet or equivalent structured input
2. a `BDC Designer` pre-experiment memo or equivalent decision output
3. an explicit kill/pass/fail gate description
4. a bounded execution scope statement
5. an evidence-status map

## What this gate is meant to prevent

This protocol exists to prevent the scientific line from:
- reopening closed directions by rhetorical drift,
- starting broad implementation before the actual question is narrow,
- confusing mechanism exploration with validated progress,
- masking missing evidence with narrative confidence,
- and overbuilding when the real gap is still analytical.

## Scientific-line interpretation rule

`BDC Designer` does not become the source of scientific truth.

The rule is narrower:

- `BDC Designer` is the first analytical gate
- direct scientific experiment remains the truth-producing layer

So the correct order is:

1. `BDC Designer` narrows the question
2. the scientific cycle executes
3. results return to a decision gate

## Required outputs by phase

### For `R1`-style physics cycles
`BDC Designer` should clarify:
- what exactly counts as structural feasibility
- what counts as collapse
- what counts as diagnostic-only success

### For `R2`-style environment cycles
`BDC Designer` should clarify:
- what environment properties are essential
- what is out of scope
- what measurements prove the task is scientifically useful

### For `R3+` mechanism cycles
`BDC Designer` should clarify:
- what mechanism is actually isolated
- what competing explanations remain open
- what evidence would justify assembly into a larger organism line

## Project rule going forward

From this point forward, a new scientific cycle is considered under-governed unless it passes through this pre-experiment gate first.

That means:

- no new cycle should start directly from intuition,
- no new phase should start from architecture desire alone,
- no new bounded mechanism cycle should start without explicit evidence-state framing.

## Definition of Done for the gate

The pre-experiment gate is considered satisfied only if:

1. the next scientific question is explicitly narrowed,
2. missing evidence is identified,
3. forbidden false directions are named,
4. execution scope is minimal and bounded,
5. direct experiment can start without ambiguity about what verdict it is meant to produce.

## Bottom line

The project now has a validated analytical discipline layer.

The scientific line should use it.

That is not product drift.

It is the correct reuse of the first successful operational result of the original BDC research program.
